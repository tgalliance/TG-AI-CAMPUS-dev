🎯 실전 프롬프트:

다음 Python 코드를 검토하고 개선점을 제시해주세요:

```python
def calculate_total(items):
    total = 0
    for i in range(len(items)):
        if items[i]['price'] > 0:
            total = total + items[i]['price'] * items[i]['quantity']
    return total
```

**검토 항목:**
1. 코드 효율성 (성능 개선)
2. 가독성 (더 명확한 표현)
3. 잠재적 버그 (예외 처리)
4. 베스트 프랙티스 (Python 관례)
5. 리팩토링 제안

**출력 형식:**
- 각 항목별 평가 (1~5점)
- 개선된 코드
- 개선 이유 설명
AI 검토 결과:

1. 코드 효율성: 3/5 - 불필요한 인덱스 사용
2. 가독성: 3/5 - 더 명확하게 표현 가능
3. 예외 처리: 2/5 - KeyError, TypeError 가능성
4. 베스트 프랙티스: 2/5 - Pythonic하지 않음

개선된 코드:
def calculate_total(items: list[dict]) -> float:
    """Calculate total price for items."""
    return sum(
        item.get('price', 0) * item.get('quantity', 0)
        for item in items
        if item.get('price', 0) > 0
    )

개선 이유:
• sum() + generator expression으로 간결하고 빠름
• get() 메서드로 KeyError 방지
• Type hints로 명확한 입출력 타입
• Docstring 추가로 문서화

최적화 코드:
@app.route('/api/orders')
def get_orders():
  orders = Order.query\
    .join(User).join(Product)\
    .with_entities(
      Order.id,
      User.name.label('user'),
      Product.name.label('product'),
      Order.total
    ).all()
  return jsonify([{
    'id': o.id,
    'user': o.user,
    'product': o.product,
    'total': o.total
  } for o in orders])

개선 효과:
• 쿼리 수: 201개 → 1개 (200배 감소)
• 응답 속도: 5초 → 0.1초 (50배 개선)
ORM을 사용할 때는 lazy loading과 eager loading의 차이를 이해하고, 실제 실행되는 SQL 쿼리를 항상 확인하세요. (SQLAlchemy의 경우 echo=True 옵션 활용)

```python
def calculate_discount(price, discount_rate, max_discount=None):
    """가격과 할인율을 받아 최종 가격을 계산합니다."""
    if not 0 <= discount_rate <= 1:
        raise ValueError("Discount rate must be between 0 and 1")
    
    discount_amount = price * discount_rate
    if max_discount and discount_amount > max_discount:
        discount_amount = max_discount
    
    return price - discount_amount
```

**요구사항:**

3. 예외 처리 테스트

from typing import List, Dict, Any

def is_eligible_user(user: Dict[str, Any]) -> bool:
    """사용자가 처리 대상인지 확인"""
    return (
        user.get('status') == 'active' and
        user.get('age', 0) >= 18 and
        user.get('country') == 'KR'
    )

def extract_user_data(user: Dict[str, Any]) -> Dict[str, Any]:
    """필요한 사용자 데이터 추출"""
    return {
        key: user[key]
        for key in ['name', 'email', 'phone', 'age']
        if key in user
    }

def process_users(users: List[Dict[str, Any]]) -> List[Dict[str, Any]]:
    """활성 성인 한국 사용자 정보 추출"""
    return [
        extract_user_data(user)
        for user in users
        if is_eligible_user(user)
    ]

✅ 중첩 if문 제거 (depth 4 → 1)
✅ 함수 분리로 단일 책임 원칙 준수
✅ List comprehension으로 Pythonic하게
✅ Type hints로 명확한 인터페이스

다음 코드에 Strategy 패턴을 적용해서 리팩토링해주세요:

```python
def calculate_shipping_cost(weight, method):
    if method == 'standard':
        return weight * 5000
    elif method == 'express':
        return weight * 10000 + 5000
    elif method == 'overnight':
        return weight * 15000 + 10000
    else:
        raise ValueError("Unknown shipping method")
```

**요구사항:**
1. Strategy 패턴 적용
2. 새로운 배송 방법 추가가 쉽게
3. 각 전략을 독립적으로 테스트 가능
4. Type hints 및 ABC 사용
from abc import ABC, abstractmethod
from typing import Dict

class ShippingStrategy(ABC):
    """배송 비용 계산 전략 인터페이스"""

    @abstractmethod
    def calculate_cost(self, weight: float) -> float:
        pass

class StandardShipping(ShippingStrategy):
    def calculate_cost(self, weight: float) -> float:
        return weight * 5000

class ExpressShipping(ShippingStrategy):
    def calculate_cost(self, weight: float) -> float:
        return weight * 10000 + 5000

class OvernightShipping(ShippingStrategy):
    def calculate_cost(self, weight: float) -> float:
        return weight * 15000 + 10000

class ShippingCalculator:
    """배송 비용 계산기"""

    _strategies: Dict[str, ShippingStrategy] = {
        'standard': StandardShipping(),
        'express': ExpressShipping(),
        'overnight': OvernightShipping()
    }

    def calculate_cost(self, weight: float, method: str) -> float:
        strategy = self._strategies.get(method)
        if not strategy:
            raise ValueError(f"Unknown shipping method: {method}")
        return strategy.calculate_cost(weight)

# 사용 예:
calculator = ShippingCalculator()
cost = calculator.calculate_cost(2.5, 'express')

장점:
✅ 새로운 배송 방법 추가 시 기존 코드 수정 불필요
✅ 각 전략을 독립적으로 테스트 가능
✅ Open-Closed Principle (OCP) 준수
✅ 런타임에 전략 변경 가능
리팩토링 시 항상 테스트 코드를 먼저 작성하세요. 기존 동작이 유지되는지 확인하지 않고 리팩토링하면 버그가 발생할 수 있습니다.

**기능:**

- 좋아요 기능
- 태그 검색

**요구사항:**
1. 모든 엔드포인트 목록 (HTTP 메서드, URL)
2. Request/Response 구조
3. 인증 방식 (JWT)
4. 에러 응답 형식
5. Pagination 방식
6. Rate Limiting 정책
📌 API 엔드포인트 설계

1. 인증:
• POST /api/v1/auth/register - 회원가입
• POST /api/v1/auth/login - 로그인 (JWT 발급)
• POST /api/v1/auth/refresh - 토큰 갱신
• POST /api/v1/auth/logout - 로그아웃

2. 게시글:
• GET /api/v1/posts - 목록 조회 (pagination, filter)
• POST /api/v1/posts - 생성 (인증 필요)
• GET /api/v1/posts/:id - 상세 조회
• PUT /api/v1/posts/:id - 수정 (작성자만)
• DELETE /api/v1/posts/:id - 삭제 (작성자만)
• POST /api/v1/posts/:id/like - 좋아요

3. 댓글:
• GET /api/v1/posts/:post_id/comments - 댓글 목록
• POST /api/v1/posts/:post_id/comments - 댓글 작성
• PUT /api/v1/comments/:id - 댓글 수정
• DELETE /api/v1/comments/:id - 댓글 삭제

4. 태그:
• GET /api/v1/tags - 태그 목록
• GET /api/v1/tags/:name/posts - 태그별 게시글

Response 형식:
{
  "success": true,
  "data": {...},
  "message": "Success",
  "pagination": {
    "page": 1,
    "per_page": 20,
    "total": 150
  }
}

Rate Limiting:
• 인증 사용자: 1000 requests / hour
• 비인증 사용자: 100 requests / hour
• 로그인 시도: 5 requests / 15 minutes

**요구사항:**

- 상품: id, name, price, image_url

**추가:**

2. 예제 쿼리
3. 장점 설명
GraphQL 스키마:
type User {
  id: ID!
  name: String!
  email: String!
  recentOrders(limit: Int): [Order!]!
}

type Order {
  id: ID!
  createdAt: String!
  totalPrice: Float!
  status: OrderStatus!
  products: [Product!]!
}

type Product {
  id: ID!
  name: String!
  price: Float!
  imageUrl: String
}

enum OrderStatus {
  PENDING
  PROCESSING
  COMPLETED
  CANCELLED
}

query {

      products {

        name
        price
        imageUrl
      }
    }
  }
}

장점:
✅ 단일 요청으로 복잡한 데이터 조회

✅ REST보다 적은 요청 횟수
✅ 강력한 타입 시스템
API 설계 시 버저닝(versioning) 전략을 미리 수립하세요. URL 버저닝(/api/v1), 헤더 버저닝, 또는 쿼리 파라미터 방식 중 선택하여 일관되게 적용하세요.
